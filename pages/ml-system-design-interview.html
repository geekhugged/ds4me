7. ML System Design

Агент ML System Design подбирает кейсы по проектированию ML-систем для подготовки к интервью (в том числе в Google): рекомендательные системы, поиск, ранжирование, детекция спама/фрода, фид-ранжирование и подобные продукты. Для каждого кейса — отдельная формулировка («Задание»: требования, ограничения, вопросы для уточнения) и подробное решение («Решение»: архитектура, метрики, оценки нагрузки/данных, trade-offs); обе части раскрываются независимо друг от друга. После вашего согласия на конкретные кейсы агент добавляет их сюда и в Streamlit-приложение.

1. Видео-рекомендации для домашней страницы (YouTube-style) Hard

Спроектируйте систему рекомендаций видео для домашней страницы крупного видеосервиса (аналог YouTube Home). Пользователь открывает приложение и видит персонализированную ленту из карточек видео; цель — максимизировать долгосрочное удовлетворение пользователя (время просмотра, возвращаемость), при этом сохраняя разнообразие и не продвигая некачественный/кликбейтный контент.

Функциональные требования:

  • На каждый запрос главной страницы вернуть ранжированный список из ~20-50 видео.
  • Учитывать историю просмотров, лайки/дизлайки, подписки, время просмотра, контекст (устройство, время суток).
  • Поддержка холодного старта — для новых пользователей и для нового контента.
  • Возможность объяснимости/diversity-ограничений (не более N видео одного канала в выдаче).

Нефункциональные требования (масштаб):

  • ~2 млрд пользователей, ~$10^9$ видео в каталоге, миллионы новых видео в день.
  • Latency бюджет на весь запрос < 200 мс (p99), часть — на клиенте/CDN.
  • Recommendation pipeline должен быть разделён на offline (batch обучение, построение индексов) и online (real-time serving) компоненты.
  • Постоянный поток новых сигналов (events) — десятки миллиардов событий в день (impressions, clicks, watch time).

Ограничения и допущения:

  • Видео имеет богатые метаданные (заголовок, описание, теги, канал, продолжительность, превью-картинка/кадры).
  • Доступна история взаимодействий пользователя за последние N дней.
  • Бизнес требует баланса между engagement-метриками и "качеством"/безопасностью контента (нельзя оптимизировать только под клики).

Вопросы для уточнения у интервьюера:

  • Какая основная бизнес-метрика — общее время просмотра, DAU/retention, выручка от рекламы или их комбинация?
  • Нужна ли персонализация в реальном времени (реакция на клики в текущей сессии) или допустимо обновление раз в N минут/часов?
  • Есть ли ограничения по разнообразию/политике (anti-clickbait, контент для детей, региональные ограничения)?
  • Какой объём исторических логов доступен для обучения, и как часто переобучаются модели?
  • Допустим ли двухстадийный pipeline (candidate generation + ranking) или нужно решение end-to-end?

1. Постановка задачи как ML-задачи

Бизнес-цель — максимизировать долгосрочное удовлетворение и удержание пользователя. В качестве ML-проксцели обычно используют комбинацию: вероятность клика $P(\text{click} \mid u, v, c)$, ожидаемое время просмотра $E[\text{watch\_time} \mid u, v, c]$ и вероятность "качественного" вовлечения (лайк, подписка, отсутствие быстрого ухода). Таргет для ранжирующей модели — взвешенная многозадачная цель (multi-task), где основной компонент — ожидаемая длительность просмотра, скорректированная штрафами за негативные сигналы (репорты, "не интересно").

Данные: implicit feedback из логов показов (impression → click/no-click → watch_time), фичи пользователя (история, демография, embedding интересов), фичи видео (метаданные, embedding контента), контекстные фичи (устройство, время, сессия).

2. Архитектура (двухстадийный pipeline)

Классическая архитектура — candidate generation → ranking → re-ranking:

  • Candidate generation (offline + online retrieval). Из миллиарда видео нужно за миллисекунды отобрать несколько сотен/тысяч кандидатов. Используется несколько источников: (1) two-tower модель (user encoder / item encoder), обученная на implicit feedback, с поиском ближайших соседей в embedding-пространстве через ANN-индекс (например, HNSW/ScaNN); (2) collaborative filtering ("похожие пользователи смотрели..."); (3) подписки и недавно загруженные видео от подписок; (4) трендовые/популярные видео по региону.
  • Ranking (scoring). Для каждого кандидата (обычно несколько сотен) вычисляется скор через более тяжёлую модель (DNN/градиентный бустинг или transformer) с богатым набором фичей: user-video cross-features, sequential модель истории просмотров (например, через RNN/Transformer по последним N видео), context. Многозадачная модель предсказывает несколько таргетов (P(click), E[watch_time], P(like), P(dislike)/skip) и комбинирует их в финальный скор.
  • Re-ranking / business rules. Применяются ограничения на разнообразие (diversity — не более K видео одного канала/категории подряд), фильтрация дублей, понижение кликбейта/некачественного контента, политика для детского контента, explore-слот для холодного старта новых видео.
  • Feature store. Offline batch jobs считают агрегированные фичи (CTR канала за 7/30 дней, среднее watch_time по категории и т.п.) и кладут их в low-latency key-value store (Redis/Bigtable) для online serving. Embedding'и пользователей и видео обновляются batch-job'ами (раз в день/часы) + near real-time обновление через streaming (Kafka/Flink) для свежих сигналов сессии.
  • Обучение и обновление модели. Ranking-модель переобучается регулярно (например, ежедневно) на скользящем окне логов с negative sampling (показы без клика — негативы). Используется warm-start от предыдущей версии. A/B-тестирование новых версий модели перед полным rollout, с мониторингом деградации (online metrics, calibration).

3. Метрики

Офлайн-метрики (на исторических логах / replay):

  • Для ranking — NDCG@k и MAP@k по релевантности (watch_time как proxy релевантности): $$ \text{NDCG@k} = \frac{DCG@k}{IDCG@k}, \quad DCG@k = \sum_{i=1}^{k} \frac{2^{rel_i}-1}{\log_2(i+1)} $$
  • Для предсказания клика/watch_time — AUC, LogLoss и калибровка предсказанной вероятности: $$ \text{LogLoss} = -\frac{1}{N}\sum_{i=1}^{N} \left[ y_i \log \hat{p}_i + (1-y_i)\log(1-\hat{p}_i) \right] $$
  • Для retrieval-стадии — Recall@k кандидатов (доля видео, которые в итоге получили высокий ranking-скор, среди отобранных candidate generation).

Онлайн/бизнес-метрики (A/B-тесты):

  • Общее время просмотра на пользователя (session watch time), CTR на главной странице.
  • Долгосрочные метрики удержания: DAU/MAU, return rate через 1/7/30 дней.
  • Метрики качества: доля "negative feedback" (dislike, hide, report), доля видео из long tail (anti-popularity-bias).

4. Оценки нагрузки и данных

  • QPS: при 2 млрд DAU и в среднем ~3 запроса главной страницы в день на активного пользователя ($\sim 5 \times 10^8$ активных в день) получаем $\approx 1.5 \times 10^9 / 86400 \approx 17{,}000$ QPS в среднем, с пиками до ~50,000-100,000 QPS.
  • Latency budget ~200мс: candidate generation (ANN-поиск среди $10^9$ embedding'ов) — ~20-30мс, ranking (скоринг ~300 кандидатов через DNN) — ~50-80мс, re-ranking и сборка ответа — ~10мс, остаток — сетевые издержки.
  • Объём логов: при $10^9$ показов в день и ~50 фичей на каждый — десятки ТБ/день необработанных логов для обучения.
  • Размер ANN-индекса: $10^9$ видео × embedding размерности 128 × 4 байта ≈ 512 ГБ — требует шардирования индекса по нескольким машинам.
  • Модель ranking: при batch-инференсе ~300 кандидатов на запрос и DNN среднего размера (~10М параметров), при 50,000 QPS это ~15М forward pass/сек — требует GPU/TPU-кластера serving с батчингом запросов.

5. Trade-offs

  • Точность vs latency. Тяжёлые sequential-модели (transformer по истории просмотров) повышают качество ranking, но увеличивают latency — решение: применять их только на стадии ranking для небольшого числа кандидатов, а не на retrieval.
  • Engagement vs "качество". Чистая оптимизация под клики/watch_time ведёт к продвижению кликбейта и feedback loop (популярные видео становятся ещё популярнее — popularity bias). Решение — multi-task с явными штрафами за негативные сигналы, регуляризация по разнообразию, отдельная модель качества контента.
  • Batch vs real-time фичи. Real-time обновление сигналов (что пользователь только что посмотрел) улучшает релевантность в сессии, но усложняет инфраструктуру (streaming pipeline) — компромисс: near real-time для критичных сигналов сессии, batch — для долгосрочных агрегатов.
  • Холодный старт. Для новых видео — explore-бонус (boosting в ранкинге на ограниченное число показов), для новых пользователей — fallback на популярные/ трендовые видео + быстрый онбординг (запрос интересов).
  • Мониторинг деградации. Регулярный мониторинг калибровки модели, дрейфа распределения фичей, доли negative feedback; shadow-режим для новых моделей перед полным rollout.

2. Предсказание CTR для рекламного аукциона Hard

Спроектируйте систему предсказания CTR (Click-Through Rate) для рекламной платформы, которая участвует в real-time аукционах за показ объявлений (аналог Google Ads / поисковой рекламы). Для каждого запроса (показа страницы) система должна оценить вероятность клика по каждому объявлению-кандидату, чтобы рассчитать ожидаемую выручку (eCPM) и определить победителя аукциона и итоговый ранкинг объявлений.

Функциональные требования:

  • На вход — запрос пользователя/контекст + набор кандидатов объявлений (от рекламодателей, прошедших таргетинг); на выход — $pCTR$ для каждого кандидата.
  • $pCTR$ используется в формуле ранжирования объявлений: $\text{rank\_score} = pCTR \times \text{bid}$ (и опционально качество объявления / quality score).
  • Модель должна быть хорошо откалибрована — предсказанная вероятность должна совпадать с реальной частотой кликов, так как от этого напрямую зависит выручка и справедливость аукциона (рекламодатель платит на основе ожидаемого CTR).
  • Поддержка холодного старта для новых объявлений/рекламодателей с минимальной историей показов.

Нефункциональные требования (масштаб):

  • Десятки/сотни тысяч аукционов в секунду глобально, на каждый — до 10-100 кандидатов объявлений, требующих pCTR.
  • Latency budget на инференс модели — единицы миллисекунд (часть общего бюджета аукциона ~50-100мс, который включает retrieval кандидатов, таргетинг, биддинг).
  • Сильный имбаланс классов — CTR обычно < 1-5%.
  • Огромная размерность категориальных фичей (миллионы уникальных query, ad_id, advertiser_id и их комбинации).

Ограничения и допущения:

  • Логи показов содержат: запрос/контекст, объявление, было ли показано, был ли клик (label), bid рекламодателя.
  • Объявления имеют разную "свежесть" — новые объявления почти не имеют истории кликов.
  • Аукцион происходит в реальном времени, модель должна быть встроена в критический путь обработки запроса.

Вопросы для уточнения у интервьюера:

  • Оптимизируем pCTR отдельно или как часть более широкой задачи (pCTR × pCVR × bid — сквозная оптимизация ценности)?
  • Какой формат фичей доступен — только табличные/категориальные или есть текстовые embedding'и запроса и объявления?
  • Какие требования к калибровке и как часто переобучаем модель (учитывая, что распределение запросов/объявлений сильно нестационарно)?
  • Допустимо ли использовать онлайн-обучение (online learning) для быстрой адаптации к новым объявлениям?
  • Как обрабатывается position bias (объявление на первой позиции кликают чаще независимо от релевантности)?

1. Постановка задачи как ML-задачи

Задача — бинарная классификация: для пары (запрос/контекст, объявление) предсказать $pCTR = P(\text{click}=1 \mid \text{query}, \text{ad}, \text{context})$. Таргет — бинарный label из логов показов (1 — был клик, 0 — не было). Ключевое требование — не просто ранжирование по AUC, а точная калибровка вероятности, так как pCTR напрямую участвует в формуле аукциона и влияет на расчёт цены за клик (например, в GSP- аукционе second-price).

2. Архитектура

  • Фичи. (1) Объявление: ad_id, advertiser_id, category, текстовый embedding заголовка/описания, историческая CTR объявления (с экспоненциальным затуханием по времени); (2) Пользователь: демография, история кликов/конверсий, embedding интересов; (3) Запрос/контекст: текст запроса (embedding), устройство, время суток, позиция показа; (4) Cross-features: query × ad category, user segment × ad category — критичны для качества и часто строятся через feature crosses или через модель, умеющую находить взаимодействия автоматически (FM/DeepFM).
  • Модель. Классический вариант — логистическая регрессия с миллионами разреженных one-hot/hashed фичей и feature crosses (быстро, интерпретируемо, легко калибровать), либо Gradient Boosted Trees (GBDT) для табличных фичей. Современный подход — Wide&Deep или DeepFM: "wide"-часть (линейная, с явными crosses) для memorization + "deep"-часть (DNN по embedding'ам категориальных фичей) для generalization. Для очень больших embedding-таблиц (миллионы ad_id) используется hashing trick для контроля размера модели.
  • Калибровка. После основной модели — отдельный калибровочный слой (Platt scaling / isotonic regression), обучаемый на hold-out выборке, чтобы $E[\hat{p} \mid \hat{p}=p] \approx p$ для каждого бакета предсказаний.
  • Pipeline. Offline: батчевое переобучение модели (например, каждые несколько часов) на скользящем окне логов показов/кликов + negative sampling (с downsampling негативов и компенсирующей перевзвешивающей калибровкой, так как класс "клик" редкий). Online: feature store с предвычисленными агрегатами (historical CTR по ad_id/advertiser, обновляется streaming-job'ом); inference-сервис в критическом пути аукциона, возвращающий pCTR за единицы мс (легковесная модель или GBDT с оптимизированным inference, например через ONNX).
  • Холодный старт. Для новых объявлений — fallback на агрегаты по advertiser_id/category вместо ad_id; explore-бюджет (показ новых объявлений с небольшим приоритетом для сбора данных); байесовское сглаживание исторической CTR ($\frac{clicks + \alpha}{impressions + \alpha + \beta}$, beta-prior по категории).

3. Метрики

Офлайн-метрики:

  • AUC / ROC-AUC — общая способность ранжировать по вероятности клика.
  • LogLoss — основная метрика для классификации с редким классом: $$ \text{LogLoss} = -\frac{1}{N}\sum_{i=1}^{N}\left[y_i \log \hat{p}_i + (1-y_i)\log(1-\hat{p}_i)\right] $$
  • Калибровка — Calibration error / Expected Calibration Error (ECE): $$ \text{ECE} = \sum_{b=1}^{B} \frac{n_b}{N} \left| \text{acc}(b) - \text{conf}(b) \right| $$ где для каждого бакета $b$ сравнивается средний предсказанный $\hat{p}$ и реальная частота кликов.
  • Normalized Cross-Entropy (NCE) относительно baseline (предсказание = средний CTR).

Онлайн/бизнес-метрики:

  • Фактический CTR показов, revenue per mille (RPM) / eCPM, выручка платформы.
  • Auction efficiency — соответствие предсказанного и фактического CTR в разрезе позиций и сегментов рекламодателей (справедливость биллинга).
  • Доля "некачественных" кликов (быстрый bounce, отсутствие конверсии) как сигнал переобучения на клик-бейт объявления.

4. Оценки нагрузки и данных

  • QPS аукционов: ~100,000 запросов/сек глобально, каждый с ~20-50 кандидатами объявлений → ~2-5 млн pCTR-предсказаний/сек на пике.
  • Latency: inference одной модели должен укладываться в ~1-2мс при батче из 20-50 кандидатов — требует легковесных моделей (GBDT с десятками деревьев или компактный DNN с квантизацией) и serving на CPU с батчингом, либо GPU для DNN-вариантов.
  • Объём логов: при $10^{10}$ показов/день и ~100 фичей на показ — сотни ГБ - единицы ТБ логов в день после агрегации.
  • Размерность фичей: миллионы уникальных ad_id/query, при hashing trick (например, $2^{24}$ bucket'ов) embedding-таблица занимает ~$2^{24} \times 16 \times 4$ байт ≈ 1 ГБ на таблицу — несколько таких таблиц для разных категориальных фичей.
  • Частота переобучения: учитывая нестационарность (новые объявления, сезонность), модель переобучается каждые несколько часов, либо используется online learning (incremental updates по мере прихода новых логов).

5. Trade-offs

  • Точность vs калибровка vs latency. Глубокие модели (DeepFM, transformer-based) дают выше AUC, но сложнее калибровать и дороже в serving — для критического пути аукциона часто выбирают GBDT или Wide&Deep с приемлемым компромиссом.
  • Position bias. Объявления на топ-позициях получают больше кликов независимо от качества — решение: включать позицию как фичу при обучении и убирать её (фиксировать на "лучшую" позицию) при инференсе, либо использовать inverse propensity weighting.
  • Имбаланс классов. Downsampling негативов ускоряет обучение, но смещает предсказанные вероятности — требуется коррекция через $q_{calibrated} = \frac{q}{q + (1-q)/w}$, где $w$ — коэффициент downsampling.
  • Холодный старт vs exploration cost. Показ новых объявлений с неопределённым CTR может снижать краткосрочную выручку, но необходим для обучения — решение через explore/exploit (например, Thompson sampling или UCB-подобные бонусы к pCTR новых объявлений).
  • Feedback loop. Модель обучается на логax показов, сделанных предыдущей моделью — объявления, которые модель никогда не показывала, не получают данных (selection bias) — частично решается через exploration traffic и off-policy correction.
  • Мониторинг. Постоянный мониторинг калибровки в разрезе сегментов (новые vs старые объявления, разные позиции), дрейфа распределения запросов, алертинг при расхождении predicted vs actual CTR.

3. Ранжирование ленты соцсети (News Feed Ranking) Medium/Hard

Спроектируйте систему ранжирования новостной ленты социальной сети (аналог Facebook/ Instagram Feed). Пользователь видит хронологически неотсортированный набор постов от друзей, групп и страниц, на которые он подписан; задача системы — отобрать и проранжировать посты так, чтобы максимизировать значимое взаимодействие пользователя (meaningful social interactions), а не просто время в приложении.

Функциональные требования:

  • Для каждого открытия ленты — вернуть ранжированный список постов из пула доступных (от подписок/друзей/групп), обычно от нескольких сотен до нескольких тысяч кандидатов на пользователя.
  • Учитывать тип контента (текст, фото, видео, реклама), автора, недавность, и предсказанную вероятность разных видов взаимодействия (like, comment, share, hide).
  • Поддержка "infinite scroll" — догрузка следующих страниц ленты с консистентным ранжированием.
  • Встраивание рекламных постов в ленту по отдельным бизнес-правилам (allocation).

Нефункциональные требования (масштаб):

  • ~1 млрд DAU, каждый открывает ленту несколько раз в день.
  • Latency p99 для построения страницы ленты < 300-500мс (несколько источников контента + ranking + ads allocation).
  • Высокая скорость появления нового контента — посты должны попадать в кандидаты в течение секунд после публикации.
  • Иерархия фич: per-user, per-post, per-author, per-edge (отношения пользователь-автор).

Ограничения и допущения:

  • Доступен социальный граф (друзья, подписки, группы) и история взаимодействий.
  • Контент крайне разнороден по типу (видео, текст, ссылки, фото) и "возрасту" (от секунд до дней).
  • Бизнес требует баланса между "engagement" и well-being метриками (не злоупотреблять сенсационным/провокационным контентом).

Вопросы для уточнения у интервьюера:

  • Какие типы взаимодействий считаются "значимыми" (meaningful) — лайк, комментарий, repost, время чтения?
  • Нужно ли учитывать недавность контента жёстко (time decay) или допустимо показывать посты "из прошлого"?
  • Как реклама встраивается в ленту — отдельный allocation-слой после ranking органического контента или единая оптимизация?
  • Есть ли требования к diversity/well-being (ограничение на количество постов одного типа/автора подряд, защита от echo chamber)?
  • Какой объём кандидатов реалистично собрать на каждый запрос (сотни vs тысячи постов)?

1. Постановка задачи как ML-задачи

Бизнес-цель — рост "значимых социальных взаимодействий" и долгосрочного удержания. ML-задача — многозадачная (multi-task) регрессия/классификация: для каждой пары (пользователь, пост) предсказать набор вероятностей $P(\text{like})$, $P(\text{comment})$, $P(\text{share})$, $P(\text{click})$, $P(\text{hide}/\text{report})$ и ожидаемое время просмотра. Итоговый скор — взвешенная сумма этих предсказаний с весами, отражающими ценность каждого типа взаимодействия (например, комментарий ценнее лайка, "hide" — сильный негативный сигнал с большим отрицательным весом). Таргеты берутся из логов показов поста в ленте (impression) и последующих действий пользователя.

2. Архитектура

  • Aggregation / Candidate generation. Из социального графа собирается пул "сырых" кандидатов — все недавние посты от друзей/подписок/групп (может быть тысячи на активного пользователя). Применяется первичная лёгкая фильтрация (уже просмотренные посты, заблокированные авторы, нерелевантные по времени) и/или лёгкая модель первого прохода (light ranking) для сокращения до нескольких сотен кандидатов.
  • Ranking (heavy model). Многозадачная DNN (например, MMoE — Multi-gate Mixture-of-Experts) принимает фичи: (1) per-user — долгосрочные интересы, embedding активности; (2) per-post — тип контента, embedding текста/изображения, возраст поста, текущая популярность (likes/comments так far); (3) per-edge (user-author affinity) — частота взаимодействий с этим автором/группой за последнее время, близость в графе; (4) контекст — время дня, устройство. Выход — набор предсказанных вероятностей/значений по каждой задаче.
  • Value model / итоговый скор. Скоры взаимодействий комбинируются в единый "value score": $$ \text{score} = \sum_{k} w_k \cdot P(\text{action}_k) $$ где веса $w_k$ задаются продуктовой командой (и могут A/B-тестироваться) — например, $w_{comment} > w_{like} > w_{click}$, $w_{hide} \ll 0$.
  • Re-ranking / Ads allocation / Diversity. После основного ranking применяются бизнес-правила: ограничение на число постов одного автора/типа подряд (diversity rules), вставка рекламных слотов по allocation-алгоритму (например, каждый N-й слот), понижение сенсационного/borderline-контента через отдельный integrity-классификатор.
  • Feature store и обновления. Edge-фичи (user-author affinity) и per-post фичи (текущее число лайков/комментариев) обновляются near real-time через streaming pipeline (Kafka/Flink), так как "свежесть" сигналов критична — пост, опубликованный минуту назад, ещё не имеет накопленной популярности, и эта информация быстро меняется. Долгосрочные user-интересы — batch-обновление раз в день.
  • Обучение. Модель переобучается регулярно (ежедневно) на логах impression → actions, с учётом position bias (позиция поста в ленте влияет на вероятность взаимодействия) — позиция как фича при обучении, фиксируется при инференсе.

3. Метрики

Офлайн-метрики:

  • Per-task AUC/LogLoss для каждой предсказываемой вероятности (like, comment, share, hide).
  • NDCG@k с релевантностью, определённой как взвешенная сумма реальных действий: $$ \text{NDCG@k} = \frac{1}{IDCG@k}\sum_{i=1}^{k} \frac{2^{rel_i}-1}{\log_2(i+1)} $$
  • Calibration error по каждой задаче (особенно для "hide"/integrity-сигналов, где важна точная вероятность для принятия решений модерации).

Онлайн/бизнес-метрики:

  • Meaningful Social Interactions (MSI) per user per day — основная North Star метрика.
  • DAU/retention (1-day, 7-day, 28-day), session length, time to first interaction.
  • Negative feedback rate (hide/report per impression), доля контента от "echo chamber" источников (diversity метрики по числу уникальных авторов в ленте).
  • Revenue per user (от ads allocation), ad load (доля рекламных постов).

4. Оценки нагрузки и данных

  • QPS: при 1 млрд DAU и ~10 открытий ленты в день на пользователя → $\approx 10^{10}/86400 \approx 116{,}000$ QPS в среднем, пики — несколько сотен тысяч QPS.
  • Кандидаты на запрос: для активного пользователя с 500+ подписками — до нескольких тысяч "сырых" кандидатов за последние 48 часов, после light ranking — ~300-500 для heavy ranking.
  • Latency budget ~300-500мс: aggregation кандидатов из графа — ~50-100мс (зависит от fan-out по подпискам), light ranking — ~30мс, heavy ranking (MMoE по ~300 кандидатам) — ~100-150мс, re-ranking/ads allocation — ~30мс.
  • Объём событий: при $10^{10}$ показов постов в день и десятках действий-таргетов на каждый — десятки ТБ логов в день.
  • Edge-фичи (user-author affinity): для 1 млрд пользователей со средним числом связей ~200-300 — граф из сотен миллиардов рёбер, каждое с набором агрегированных счётчиков — требует распределённого key-value хранилища с обновлением near real-time.

5. Trade-offs

  • Engagement vs well-being. Чистая оптимизация под клики/время в приложении продвигает сенсационный контент — решение: multi-task с явными негативными весами для "hide"/"report" и отдельный integrity-слой, понижающий borderline-контент независимо от предсказанного engagement.
  • Свежесть vs точность ranking-фичей. Near real-time обновление счётчиков популярности поста важно для свежего контента, но усложняет инфраструктуру — компромисс: streaming для критичных edge/post сигналов, batch — для долгосрочных user-интересов.
  • Diversity vs релевантность. Жёсткие diversity-правила (не более 2 постов одного автора подряд) могут понижать топ-кандидатов с самым высоким скором — обычно реализуется как мягкое штрафование (penalty), а не hard filter.
  • Сложность модели vs latency. MMoE с множеством задач даёт лучшее качество, но дороже в инференсе — для light ranking первого прохода используется существенно упрощённая модель (логистическая регрессия/маленькое дерево) на большем числе кандидатов.
  • Position bias и feedback loop. Посты, показанные на низких позициях, реже получают взаимодействия и реже попадают в положительные обучающие примеры — корректируется через позицию как фичу и периодическое exploration (случайное "встряхивание" порядка для части трафика).
  • Мониторинг. Дрейф распределения контента (новые форматы — Reels, Stories), деградация per-task калибровки, регулярные A/B на изменения весов $w_k$ итоговой формулы скоринга.

4. Детекция фрода в платёжных транзакциях Hard

Спроектируйте систему детекции фрода для платёжной платформы (аналог системы, проверяющей каждую транзакцию по карте в момент оплаты). Для каждой транзакции система должна в реальном времени принять решение: разрешить, заблокировать или отправить на дополнительную проверку (например, 3D-Secure / ручной review), минимизируя финансовые потери от фрода при сохранении низкого уровня false positives (блокировка легитимных платежей вредит бизнесу и пользовательскому опыту).

Функциональные требования:

  • На вход — транзакция (сумма, мерчант, карта/аккаунт, устройство, геолокация, время) + история предыдущих транзакций пользователя/карты; на выход — скор фрода и решение (allow/challenge/block).
  • Решение принимается синхронно, в момент авторизации платежа.
  • Поддержка адаптации к новым видам фрода (fraudsters постоянно меняют тактики — adversarial environment).
  • Объяснимость решений для compliance и для ручного review команды.

Нефункциональные требования (масштаб):

  • Десятки тысяч транзакций в секунду на пике (например, Black Friday).
  • Latency p99 для скоринга < 100мс (часть бюджета авторизации платежа, который обычно ограничен ~1-2 секундами end-to-end).
  • Крайне сильный имбаланс классов — фрод составляет обычно < 0.1-1% транзакций.
  • Требуется низкая latency для построения признаков на основе истории (агрегаты за последние минуты/часы — "сколько транзакций с этой карты за последний час").

Ограничения и допущения:

  • Label приходит с задержкой — фрод обычно подтверждается через чарджбэк/жалобу пользователя спустя дни/недели (delayed feedback).
  • Фродстеры активно адаптируются к детектору (adversarial, концепт-дрифт).
  • Стоимость false positive (блокировка легитимного платежа) и false negative (пропущенный фрод) сильно различается по величине и должна учитываться явно.

Вопросы для уточнения у интервьюера:

  • Какие действия доступны помимо allow/block — есть ли промежуточный "step-up authentication" (3DS, OTP)?
  • Как учитывается asymmetric cost — есть ли явная денежная оценка FP vs FN, чтобы выбрать порог решения?
  • Какая задержка допустима для получения labels (chargeback period) и как это влияет на цикл переобучения?
  • Нужна ли отдельная rule-based система как safety net поверх ML-модели (для известных паттернов фрода с явными правилами)?
  • Есть ли доступ к графу связей (общие устройства/карты/адреса между аккаунтами) для выявления organized fraud rings?

1. Постановка задачи как ML-задачи

Задача — бинарная классификация с крайне несбалансированными классами: для каждой транзакции предсказать $P(\text{fraud}=1 \mid x)$, где $x$ — фичи транзакции, истории и контекста. Таргет — бинарный label, получаемый из подтверждённых чарджбэков/жалоб (положительный, но с задержкой в днях/неделях) и неоспоренных транзакций спустя период наблюдения (отрицательный). Финальное решение — не просто классификация, а выбор действия (allow/challenge/block) на основе скора и калиброванной по cost-matrix пороговой политики.

2. Архитектура

  • Real-time feature pipeline. Помимо статических фичей транзакции (сумма, мерчант, MCC-код, страна, устройство), критичны velocity-фичи — агрегаты по истории за короткие окна: число транзакций с этой карты за последние 5 мин/1 час/24 часа, сумма транзакций за период, число уникальных мерчантов/стран за последний час и т.п. Эти агрегаты считаются в streaming-сервисе (Kafka + Flink/Storm) с хранением состояния в low-latency store (Redis), чтобы быть доступными за <10мс на момент скоринга.
  • Многоуровневый scoring pipeline. (1) Rule-based слой — быстрые жёсткие правила для известных паттернов (например, "более 5 транзакций за 1 минуту с одной карты" → block немедленно), служит safety net и снижает нагрузку на ML-модель; (2) ML-модель — основной скоринг: Gradient Boosted Trees (XGBoost/LightGBM) — стандарт для табличных фичей с имбалансом, хорошо работает с разреженными категориальными фичами и легко интерпретируется (feature importance, SHAP); опционально — graph-based модель (GNN или граф-фичи) для обнаружения колец фрода через связи карта-устройство-адрес.
  • Принятие решения. Скор модели сравнивается с порогами, выбранными на основе cost-matrix: $\text{threshold}$ минимизирует ожидаемую стоимость $E[\text{cost}] = C_{FP} \cdot P(FP) + C_{FN} \cdot P(FN)$. Обычно несколько порогов — низкий скор → allow, средний → step-up authentication (3DS/OTP), высокий → block + ручной review для high-value транзакций.
  • Обработка delayed feedback. Так как окончательный label приходит через недели, используется комбинация: (1) обучение на "созревших" данных с лагом (например, транзакции старше 60 дней с финальным label); (2) промежуточные слабые сигналы (отказ от платежа на этапе 3DS, ручной review результат) как дополнительные сигналы для более быстрой адаптации; (3) периодическая переоценка исторических предсказаний по мере поступления новых chargebacks.
  • Обработка имбаланса. Undersampling негативного класса при обучении + калибровка предсказаний (Platt scaling) для восстановления реальных вероятностей; либо class weights / focal loss в GBDT/DNN.
  • Адаптация к concept drift. Частое переобучение (например, ежедневно) на скользящем окне, мониторинг распределения фичей и скоров для раннего обнаружения новых паттернов фрода, A/B-тест новых версий моделей в shadow-режиме перед полным rollout.

3. Метрики

Офлайн-метрики:

  • PR-AUC (Precision-Recall AUC) — предпочтительнее ROC-AUC при сильном имбалансе, так как ROC-AUC может быть обманчиво высоким.
  • Precision и Recall при заданном пороге: $$ \text{Precision} = \frac{TP}{TP+FP}, \quad \text{Recall} = \frac{TP}{TP+FN} $$
  • $F_\beta$-score с $\beta > 1$ (приоритет recall, так как пропущенный фрод обычно дороже false positive): $$ F_\beta = (1+\beta^2) \cdot \frac{\text{Precision}\cdot\text{Recall}}{\beta^2 \cdot \text{Precision} + \text{Recall}} $$
  • Calibration — насколько предсказанная вероятность соответствует реальной частоте (важно для cost-based threshold'ов).

Онлайн/бизнес-метрики:

  • Fraud loss rate (доля выручки, потерянной на фроде, $bps$ от объёма транзакций).
  • False positive rate на легитимных транзакциях (доля заблокированных/challenged легитимных платежей — влияет на conversion и customer satisfaction).
  • Approval rate (доля одобренных транзакций) — бизнес обычно требует не снижать его значительно при внедрении новой модели.
  • Time-to-detection для новых паттернов фрода (как быстро система адаптируется к новой атаке).

4. Оценки нагрузки и данных

  • QPS: на пике (Black Friday/Cyber Monday) — десятки тысяч транзакций/сек, например 50,000 TPS; в обычные дни — на порядок ниже.
  • Latency budget ~100мс: получение velocity-фичей из Redis — ~5-10мс, rule-based проверки — ~1-2мс, GBDT-инференс (сотни деревьев, ~200 фичей) — ~5-15мс, агрегация решения — ~5мс, остаток — сетевые издержки и запас.
  • Имбаланс: при 0.5% фрода и $10^9$ транзакций/месяц — ~5М фродовых транзакций/месяц в обучающей выборке (после агрегации с разных мерчантов) — достаточно для обучения, но требует careful sampling.
  • Velocity-агрегаты: для каждой активной карты/аккаунта ($10^8$+ активных карт) хранится набор счётчиков за разные окна (5 мин/1ч/24ч/7д) — десятки счётчиков на карту, итого сотни ГБ в low-latency хранилище с TTL-эвикцией.
  • Объём данных для обучения: за 60 дней лага для созревания labels — нужно хранить историю транзакций минимум 60-90 дней (с учётом периодов рассмотрения чарджбэков), десятки ТБ.

5. Trade-offs

  • Precision vs Recall (cost-sensitive threshold). Снижение порога увеличивает recall (меньше пропущенного фрода), но растит FP (больше легитимных платежей блокируется) — порог выбирается через явную cost-matrix, может различаться по сегментам (high-value транзакции — более консервативный порог).
  • Latency vs богатство фичей. Velocity-фичи и графовые признаки сильно повышают качество, но требуют дорогой real-time инфраструктуры — компромисс: базовый набор фичей доступен мгновенно из кэша, более сложные графовые фичи — асинхронно обновляются и используются с небольшим лагом.
  • Adversarial adaptation. Фродстеры изучают паттерны детектора — частое переобучение и rule-based safety net помогают, но создают риск overfitting на исторические атаки и упущения новых — необходим постоянный мониторинг новых кластеров аномалий (unsupervised anomaly detection как дополнение к supervised модели).
  • Delayed feedback vs скорость адаптации. Ожидание "созревших" labels (60+ дней) обеспечивает качество обучающих данных, но замедляет реакцию на новые атаки — решение: гибридный подход с быстрыми слабыми сигналами (отказы 3DS, ручной review) для оперативного дообучения/правил.
  • Интерпретируемость vs сложность модели. GBDT с SHAP-объяснениями предпочтительнее глубоких сетей для compliance и ручного review — несмотря на потенциально более высокое качество DNN/GNN на графовых данных.
  • Мониторинг. Дашборды по distribution drift фичей и скоров, алертинг при резком росте fraud rate или approval rate в отдельных сегментах (страна/мерчант/BIN карты), регулярные ручные аудиты false negatives через выборочный review одобренных транзакций.

5. Визуальный поиск похожих товаров (e-commerce) Medium

Спроектируйте систему визуального поиска для маркетплейса: пользователь загружает фото товара (например, сфотографировал понравившуюся вещь на улице или скриншот из соцсети), а система возвращает похожие товары из каталога (аналог Google Lens Shopping / Pinterest Visual Search / "Shop the look").

Функциональные требования:

  • На вход — изображение (фото/скриншот), возможно с указанием bounding box интересующего объекта; на выход — ранжированный список из топ-K похожих товаров каталога с фотографиями и метаданными (цена, ссылка).
  • Похожесть должна учитывать визуальное сходство (форма, цвет, текстура, стиль), а не только точное совпадение изображений.
  • Поддержка multi-object изображений (детекция и поиск по отдельным объектам на фото — например, отдельно по сумке и по обуви на одной фотографии).
  • Каталог постоянно обновляется — новые товары должны становиться доступными для поиска в течение разумного времени (часы, не недели).

Нефункциональные требования (масштаб):

  • Каталог — сотни миллионов товаров с изображениями.
  • Latency p99 на запрос < 500мс-1с (пользователь ждёт результат интерактивно).
  • QPS — от тысяч до десятков тысяч запросов в секунду в пиковые периоды (распродажи).
  • Изображения пользователей сильно отличаются по качеству/ракурсу/фону от "чистых" студийных фото каталога (domain gap).

Ограничения и допущения:

  • Для каждого товара каталога доступно одно или несколько изображений + текстовые метаданные (категория, бренд, атрибуты).
  • Нет явных labels "похожести" между товарами — нужно либо использовать proxy-сигналы (совместные просмотры/покупки), либо self-supervised обучение embedding'ов.
  • Возможна разметка через crowdsourcing для оценки качества, но в ограниченном объёме.

Вопросы для уточнения у интервьюера:

  • Нужна ли детекция объектов на фото (multi-object) или предполагается, что пользователь уже выделил/сфотографировал один товар?
  • Что считается "похожим" — визуально идентичный товар (same item retrieval) или стилистически похожий (similar style/category)?
  • Какие сигналы доступны для обучения embedding'ов — пары "просмотрено вместе", категории/атрибуты, или нужно идти через pure self-supervised (contrastive learning на изображениях)?
  • Есть ли требования к real-time индексации новых товаров или допустим batch-обновление индекса раз в день?
  • Как оценивать качество — через offline-разметку (human eval похожести) или через online-метрики (click-through на результаты визуального поиска)?

1. Постановка задачи как ML-задачи

Задача — image retrieval: научиться отображать изображения товаров в общее embedding-пространство так, чтобы визуально/стилистически похожие товары были близки по косинусному расстоянию, а непохожие — далеки. Это задача representation learning, обучаемая через contrastive/metric learning (triplet loss, contrastive loss или более современные подходы — CLIP-style или supervised contrastive с категориями как weak labels). На инференсе запрос — это embedding изображения пользователя, поиск — k-NN в индексе embedding'ов каталога.

2. Архитектура (двухстадийный pipeline)

  • Object detection (опционально, для multi-object). Если изображение может содержать несколько товаров, сначала применяется детектор объектов (например, YOLO/Faster R-CNN, дообученный на категориях каталога — "сумка", "обувь", "одежда"), возвращающий bounding box'ы кандидатов. Каждый bbox обрабатывается отдельно как независимый запрос.
  • Feature extraction (embedding model). CNN или Vision Transformer (например, ResNet/EfficientNet/ViT, дообученный или с CLIP-инициализацией) преобразует изображение (или crop по bbox) в dense embedding (например, 512-1024 измерений). Обучение — metric learning: triplet loss с anchor-positive-negative парами, где positive — тот же товар с другого ракурса/фото или товар из того же "купили вместе" кластера, negative — случайный товар другой категории. Hard negative mining критичен для качества (negatives из похожей категории, но другого товара).
  • ANN-индекс (candidate retrieval). Embedding'и всех товаров каталога (сотни миллионов) индексируются в приближённом nearest-neighbor индексе (FAISS/ScaNN/ HNSW) с использованием квантизации (например, product quantization) для уменьшения размера индекса. На запрос — top-K (например, 100) ближайших по косинусному расстоянию.
  • Re-ranking. Top-K кандидатов от ANN дополнительно ранжируются более точной (но дорогой) моделью — например, cross-attention модель, сравнивающая запрос и кандидата напрямую, или комбинация визуального скора с текстовыми/категориальными сигналами (совпадение категории, бренда, ценового диапазона) и бизнес-сигналами (популярность, наличие на складе, маржинальность).
  • Индексация каталога. Offline batch pipeline: при добавлении нового товара — извлечение embedding'а через feature extractor, добавление в ANN-индекс. Полная переиндексация (rebuild) — периодически (например, еженедельно) для оптимизации структуры индекса; incremental добавления — near real-time (новые товары появляются в поиске в течение часов).
  • Domain gap (фото пользователя vs студийные фото). Для сокращения разрыва между "грязными" пользовательскими фото (плохое освещение, фон, ракурс) и чистыми каталожными изображениями — аугментации при обучении (имитация плохих условий: случайный crop, изменение яркости/контраста, добавление фона), либо отдельная query-side модель, дообученная на парах "пользовательское фото — каталожное фото того же товара" (если такие пары собираются через crowdsourcing).

3. Метрики

Офлайн-метрики:

  • Recall@k и Precision@k на размеченном датасете "запрос → релевантные товары": $$ \text{Recall@k} = \frac{|\{\text{релевантные товары в top-k}\}|}{|\{\text{всего релевантных товаров}\}|} $$
  • mAP (mean Average Precision) — усреднённая точность по всем запросам с учётом ранжирования релевантных результатов.
  • Качество embedding-пространства — separability через triplet accuracy (доля триплетов, где $d(\text{anchor}, \text{positive}) < d(\text{anchor}, \text{negative})$).

Онлайн/бизнес-метрики:

  • CTR на результаты визуального поиска (доля запросов, по которым пользователь кликнул хотя бы на один результат).
  • Conversion rate от визуального поиска (доля, завершившаяся покупкой), сравнение с текстовым поиском.
  • Latency p50/p95/p99 end-to-end (detection + embedding + ANN + re-ranking).
  • Coverage — доля запросов, для которых найден хотя бы один результат с similarity выше порога.

4. Оценки нагрузки и данных

  • Размер каталога: $5 \times 10^8$ товаров, embedding размерности 512 × 4 байта (float32) ≈ 1 КБ на товар → "сырой" объём embedding'ов ≈ 500 ГБ; с product quantization (например, сжатие до 64 байт на вектор) — индекс уменьшается до ~32 ГБ, что позволяет держать его в памяти на нескольких машинах.
  • QPS: 10,000-50,000 запросов/сек в пик; каждый запрос — 1+ изображений (multi-object), в среднем ~1.5 объекта на изображение → ~75,000 embedding extraction/сек на пике.
  • Latency budget ~500мс-1с: object detection — ~50-100мс, feature extraction (CNN/ViT inference) — ~50-100мс на GPU с батчингом, ANN-поиск (top-100 из $5\times10^8$) — ~10-30мс при шардированном индексе, re-ranking top-100 → top-20 — ~50-100мс, остаток — сетевые издержки.
  • Инфраструктура для feature extraction: при 75,000 inference/сек и ~10мс на inference (батч на GPU) — требуется десятки GPU-инстансов с батчингом запросов (dynamic batching).
  • Обновление индекса: при добавлении ~1М новых товаров/день — incremental добавление в ANN-индекс near real-time, полный rebuild (для оптимальной структуры индекса) — еженедельно в offline batch job, занимающий часы на распределённом кластере.

5. Trade-offs

  • Same-item vs similar-style retrieval. Модель, обученная находить "точно такой же товар", может плохо находить "стилистически похожие" товары и наоборот — часто решается через несколько embedding-голов (multi-task) или отдельные индексы для разных типов поиска ("найти точно этот товар" vs "найти похожие").
  • Точность ANN vs latency/память. Точный k-NN невозможен при $5\times10^8$ векторах — приближённые методы (HNSW, IVF+PQ) дают recall ~90-95% от точного поиска при десятках мс, что является приемлемым trade-off.
  • Domain gap. Без аугментаций/доменной адаптации модель, обученная на студийных фото, плохо работает на "грязных" пользовательских фото — критично инвестировать в аугментации и/или сбор пар "user photo — catalog photo" того же товара.
  • Размер embedding vs качество. Большая размерность embedding (1024+) даёт лучшее качество, но увеличивает размер индекса и latency поиска — компромисс через dimensionality reduction (PCA) после обучения или дистилляцию в меньшую модель.
  • Холодный старт новых товаров. Embedding для нового товара доступен сразу после прогона через feature extractor (нет проблемы "нет истории взаимодействий", как в CF-based рекомендациях) — но popularity-based re-ranking сигналы (продажи, просмотры) отсутствуют первое время, что можно компенсировать explore-бонусом в re-ranking.
  • Мониторинг. Дрейф распределения запросов (новые типы товаров/трендов), мониторинг coverage (доля запросов без релевантных результатов), регулярный human eval выборки результатов для отслеживания деградации качества embedding-модели после обновлений каталога.
